# 견적요청 중복 기재 원인과 해결 방안

---

## 1. 중복 기재가 발생한 원인

### 배경

GanpoomTracker를 처음 붙일 때 airbridge.events.send 를 가로채는 래퍼를 index.html 에 설치했다.
airbridge 이벤트가 발생할 때마다 GanpoomTracker도 자동으로 같이 호출되도록 하기 위함.

```
airbridge.send() 호출
  └→ 래퍼가 가로챔
       └→ GanpoomTracker.track() 자동 실행
```

### 래퍼의 타이밍 문제로 실제 견적 누락 발생

래퍼는 airbridge 와 GanpoomTracker 가 **둘 다 로드된 후**에야 적용된다.
gp.js 가 `async` 로 로드되기 때문에, 페이지 초기에는 래퍼가 적용되지 않은 구간이 존재한다.

```
페이지 로드
  ├─ airbridge 로드 완료
  │    └─ 래퍼 미적용 상태 (gp.js 아직 로딩 중)
  │         └─ 이 구간에 견적 제출 → GanpoomTracker 에 기록 안 됨 (누락)
  └─ gp.js 로드 완료 → 래퍼 적용
```

실제로 이 타이밍 문제로 인해 고객 견적요청이 누락되는 사례가 발생했다.

### 누락 방지를 위해 직접 호출 추가

누락을 막기 위해 중요한 견적요청 이벤트에 GanpoomTracker.track() 직접 호출을 추가했다.

```javascript
window.airbridge.events.send('comparison.request', { action })          // airbridge 전송
if (window.GanpoomTracker) window.GanpoomTracker.track('comparison.request', {})  // 안전장치
```

직접 호출은 airbridge.send 다음 줄에서 실행되기 때문에, gp.js 가 그 사이에 로드 완료되면
래퍼가 적용되지 않았더라도 직접 호출로 GanpoomTracker 에 기록할 수 있다.

### 결과적으로 중복이 발생하는 구조

래퍼가 적용된 상태에서 견적이 제출되면:

```
airbridge.send() 호출
  └─ 래퍼 → GanpoomTracker.track() 자동 실행   ← 1번째
직접 호출 → GanpoomTracker.track()              ← 2번째
```

→ 동일 견적이 2번 기록됨

래퍼 적용 여부가 네트워크 상태나 캐시에 따라 달라지기 때문에 **간헐적으로** 중복이 발생했다.

## 2. 중복을 막기 위해 시도한 것

### 시도 1 — 서버 dedup (SELECT → INSERT)

견적 이벤트가 서버에 도착할 때, 5초 이내에 동일한 session_id + event_category 가 있으면 저장하지 않는 방식.

**문제점 — Race Condition**
```
요청 A 도착 → DB 조회: "없네" ──────────────────────→ 저장
요청 B 도착 → DB 조회: "없네" (A 아직 저장 전) → 저장
```
12ms 간격으로 두 요청이 거의 동시에 오면 SELECT 시점에 둘 다 "없다"고 판단해서 둘 다 저장됨.

### 시도 2 — DB unique constraint + upsert (실패)

DB 컬럼에 unique index 를 걸고, 중복 시 에러 없이 무시하는 upsert 방식으로 교체 시도.

**실패 원인**
Partial Index (`WHERE dedup_key IS NOT NULL`) 를 사용했는데,
Supabase(PostgREST) 가 이를 `ON CONFLICT` 에서 인식하지 못해 400 에러 발생.
견적 기재가 아예 안 되는 상황 발생 → **롤백 처리함**

## 3. 결론 및 올바른 해결 방안

### 왜 클라이언트 코드 수정만으로는 해결할 수 없는가

- 직접 호출을 제거하면 실제로 발생했던 누락 문제가 다시 생길 수 있음
- 래퍼의 타이밍 문제는 구조적인 한계이며, 완전히 없애려면 gp.js 로딩 방식을 바꿔야 함
- 중복이 발생하더라도 서버에서 안전하게 1번만 기록하는 것이 현실적인 해결책

### 올바른 서버 dedup 적용

이번 실패의 원인은 Partial Index 였고, 일반 Unique Index 로 바꾸면 해결된다.

PostgreSQL 에서 NULL 은 NULL 끼리 서로 다른 값으로 처리된다.
따라서 견적이 아닌 이벤트들(dedup_key = NULL)은 충돌 없이 모두 정상 저장된다.

```sql
-- 기존 (잘못된 Partial Index → 400 에러 원인)
CREATE UNIQUE INDEX events_dedup_key_idx ON events (dedup_key) WHERE dedup_key IS NOT NULL;

-- 올바른 방법 (일반 Unique Index)
DROP INDEX IF EXISTS events_dedup_key_idx;
CREATE UNIQUE INDEX events_dedup_key_idx ON events (dedup_key);
```

이렇게 하면:
- 견적요청이 2번 들어와도 → DB 가 원자적으로 1번만 저장
- 누락 가능성 없음 (둘 다 실패하는 경우 없음)
- Race condition 없음 (SELECT→INSERT 방식과 달리 DB 레벨에서 처리)

---

## 4. 현재 상태

| 항목 | 상태 |
|------|------|
| 서버 dedup (SELECT→INSERT) | 유지 중 (race condition 있으나 대부분 정상 작동) |
| DB unique constraint | 롤백됨 (dedup_key 컬럼은 DB에 남아있으나 인덱스 미적용) |
| 클라이언트 직접 호출 | 유지 중 (누락 방지를 위해 필요한 코드) |
